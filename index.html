B2B SaaS ERG / Community Enterprise

Audience targeting for event & community organizers

Giving event creators the ability to notify exactly the right people — not everyone — when publishing events inside an enterprise community platform.

My role
Lead Designer
Scope
End-to-end feature design
Platform
Enterprise SaaS (web)
Users
Event & ERG organizers
🎯

The problem

No control over who gets notified

Event creators on the platform had a blunt tool: publish an event, notify everyone. There was no way to control who received the notification — which was a real problem inside ERG (Employee Resource Group) communities where events are intentional and audience-specific.

"For a Women in Leadership event, we should be able to target people in leadership or management roles — not blast the entire organization."

The original trigger was even narrower: a request to filter event notifications by geographic location so location-specific events only reached relevant people. But as conversations evolved with PMs and stakeholders, it became clear location alone wasn't enough — the need was much broader.

Discovery

From one filter to a flexible system

What started as "location filtering" expanded into a full audience targeting system. Organizers needed to filter by:

  • Location — notify people in a specific office or region
  • Department — target by team or org unit, with multi-select
  • Title / role — reach leadership, management, or specific seniority
  • Individual names — hand-pick specific people
  • CSV import — bulk upload for large or pre-defined audience lists

ERG community leaders were the primary users — they were already creating focused, theme-specific events and wanted the ability to invite the right subset without manual workarounds.

Design challenges

The hard problems

  • AND/OR filter logic — organizers needed "users in NY and Engineering" as well as "users in NY or Engineering." Designing this to feel intuitive without exposing raw logic operators was the core UX challenge.
  • Post-publish audience — once an event is live, organizers work from the event dashboard, not the creation form. Reopening the form risked confusion and duplicate invites.
  • CSV error handling — bulk imports inevitably fail on some rows. The design needed to handle errors gracefully: show which rows failed, why, and let users download and fix them.
  • Auto-register discoverability — an existing feature clients kept asking for because they couldn't find it. The toggle was buried in the confirmation modal and easy to miss.

Solution

What we shipped

Audience modal with filter groups

Filter by location, title, department, or individual name. Organizers can add and remove filter items and groups, with AND/OR logic surfaced in plain language rather than operator symbols.

CSV import with error handling

Alternative to manual filtering for large or pre-defined lists. Failed rows are shown with a reason, downloadable as a fixed template, and replaceable without starting over.

Post-publish audience actions

Additional audience actions surfaced via a kebab menu in the event dashboard — kept separate from the creation form to avoid confusion. Duplicate invites are prevented automatically.

Unified communication flow

Replaced scattered entry points with a single guided flow. One button surfaces all options: invite, auto-register, send reminder, invite all subscribers, register all subscribers. Reduced support requests about "missing" features.

Outcome

Results & reflections

No formal KPI data at time of publication, but ERG communities adopted audience targeting immediately for group-specific events — which was the core use case the feature was built for.

The improved communication flow addressed a recurring client misconception that auto-registration didn't exist, which had generated ongoing support requests.

If I were to do it again, I'd push for a lightweight usability test on the AND/OR filter UI before shipping — the logic interaction is the riskiest part of the design and the one most worth validating early.

Post-launch monitoring on which communication actions are used most would also inform future prioritization.
